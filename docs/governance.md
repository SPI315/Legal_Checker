# Governance: Legal Doc Checker (PoC)

## 1. Risk Register

| Риск | Вероятность/влияние | Детект | Защита | Остаточный риск |
|---|---|---|---|---|
| Утечка персональных данных во внешний LLM/API | Средняя / Критичное | Аудит payload до вызова, DLP-проверка логов | Обязательная анонимизация перед внешними вызовами, запрет отправки сырого текста | Низкий-средний |
| Prompt injection из текста документа | Высокая / Высокое | Red-team тесты на инъекции, аномальные ответы модели | Документ трактуется как данные, жёсткий system prompt, output validation | Средний |
| Ошибки OCR на сканах приводят к пропуску рисков | Высокая / Среднее | OCR quality score, низкая уверенность/пустые секции | Локальный OCR + fallback пайплайн, пометка о неопределенности в отчете | Средний |
| Галлюцинации/ложные выводы LLM | Средняя / Высокое | Выборочная экспертная проверка | Rule-based cross-check, цитаты и ссылки на источники, оценка уверенности | Средний |
| Устаревшая или нерелевантная нормативная информация из web-поиска | Средняя / Высокое | Проверка даты/источника, ручная валидация выборки | Сохранение timestamp источников, указание ограничений в UI | Средний |
| Ложные совпадения в локальном RAG (не тот документ/версия) | Средняя / Среднее | Трассировка источников (doc_id/chunk_id) | Версионирование базы данных, threshold на уверенность | Низкий-средний |
| Небезопасные tool-calls/обработка вредоносных файлов | Низкая-средняя / Высокое | Логи tool-calls, алерты по нарушениям политик | Проверка MIME/расширений, политики использования tools | Низкий |

## 2. Политика логов и персональных данных

- В логах и БД запрещено хранить полный исходный текст документа.
- Разрешено хранить:
  - `anonymized_text`;
  - `spans` (типы сущностей и offsets для подсветки);
  - `analysis_report` (ссылки на источники без раскрытия PII);
  - технические метрики и статусы этапов.
- Хранение оригинального текста допускается только in-memory на время обработки запроса.
- Политика хранения PoC: удаление и управление артефактами вручную.
- Все LLM-вызовы логируются с информацией по токенам и стоимости. Предварительная схема: provider/model, prompt/completion tokens, estimated cost, document_id, stage, timestamp.

## 3. Политика безопасного вызова инструментов и интеграций

- Local-first OCR: OCR выполняется локально; внешние OCR/VLM-сервисы не используются по умолчанию.
- VLM fallback: допускается только как явный fallback-режим и только при соблюдении политики PII.
- Ограничение источников для проверки соответствия нормативным требованиям доверенными доменами.
- Валидация входных файлов: MIME/расширение.

## 4. Защита от injection и подтверждение действий

- Prompt injection:
  - системная инструкция с запретом исполнять инструкции из документа;
  - очистка/разделение служебного и пользовательского контекста;
  - пост-валидация.
- Action confirmation:
  - в UI должны отображаться источник вывода и уровень уверенности.

## 5. Evals, мониторинг, failover

- Evals:
  - внутренний набор документов с эталонной разметкой;
  - регулярный расчет `Precision@Top5` и процента ошибочных ссылок на документы.
- Monitoring:
  - p95 latency end-to-end и по этапам;
  - error rate по модулям;
  - OCR success rate;
  - token/cost dashboard.
- Failover:
  - при недоступности LLM сервис возвращает rule-based риски с пометкой;
  - retry/backoff и таймауты на внешних интеграциях.
