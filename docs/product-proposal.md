# Product Proposal: Legal Doc Checker (PoC)

## 1. Обоснование идеи

Юридические команды тратят значительное время на первичную проверку договоров и других документов, при этом типовые риски регулярно требуют повторяемой, рутинной проверки. Цель PoC — автоматизировать именно этот первичный слой контроля: извлечение текста, анонимизацию персональных данных, обнаружение рисков и разметку проблемных фрагментов с объяснением.

Прикладная ценность:

- сокращение времени первичного разбора документа;
- повышение качества обнаружения типовых рисков;
- стандартизированная и объяснимая разметка для последующей валидации юристом.

## 2. Цель и метрики

### 2.1 Продуктовая цель

Повысить качество первичного выявления рисков в юридических документах для юриста.

### 2.2 Продуктовые метрики

- Основная метрика качества: `Precision@Top5 >= 0.80` на внутреннем тестовом наборе - доля документов, где в Top-5 есть минимум 1 релевантный риск.
- Среднее время до первого найденного риска в UI.

### 2.3 Агентские метрики

- Доля рисков с корректной привязкой к нужному абзацу/диапазону.
- Доля рисков с валидным evidence (цитата + paragraph_id).
- Доля запросов к LLM без утечек сырых данных (целевое значение: 100%).

### 2.4 Технические метрики

- `p95 latency <= 60s`.
- Error rate по пайплайну (парсинг/OCR/анонимизация/анализ/экспорт).
- OCR text extraction success rate для PDF-сканов.

## 3. Сценарии использования и edge-кейсы

### 3.1 Основной сценарий

1. Юрист загружает `docx`.
2. Система парсит документ в абзацы с перекрытием.
3. Система анонимизирует чувствительные данные.
4. Система анализирует документ и находит риски.
5. UI показывает найденные риски с подсветками и комментариями.
6. Пользователь экспортирует JSON и/или `docx` с комментариями.

### 3.2 Ключевые типы рисков MVP

- Пролонгация без согласования.
- Односторонний отказ от ответственности.
- Одностороннее изменение условий.

### 3.3 Edge-кейсы

- PDF-скан плохого качества: OCR возвращает шумный текст.
- Частично поврежденный `docx` или нестандартная структура документа.
- Смешение языков и нестандартные юридические формулировки.
- Ложные срабатывания анонимизации.
- Недоступность внешнего LLM/API.
- Конфликт между рекомендацией LLM и локальной нормативной/проектной базой.

## 4. Ограничения

### 4.1 Технические ограничения

- Целевой SLO для PoC: p95 до 60 секунд на документ среднего объема.
- Для OCR используется локальный движок; качество зависит от качества скана.
- На старте допускается неидеальное извлечение структуры для сложных PDF.

### 4.2 Операционные ограничения

- Явные бюджетные лимиты на старте не фиксируются.
- Обязательно ведется детальный учет токенов и стоимости по каждому LLM-вызову:
  - `document_id`, этап пайплайна, модель, токены, стоимость, timestamp.
- Политика хранения: оригинальный текст допускается только in-memory во время обработки; хранение и удаление артефактов — вручную.

## 5. Архитектурный набросок

### 5.1 Модули

- Frontend UI: загрузка, просмотр абзацев, список найденных рисков, подсветка в исходном тексте, экспорт.
- Backend API (FastAPI): оркестрация пайплайна и state.
- Document Parser: `docx`/`pdf` парсинг с перекрытием, сегментация параграфов.
- OCR модуль (локальный): извлечение текста из сканов PDF.
- PII Anonymizer: маскирование PII и формирование spans (замещающие метки).
- Analyzer:
  - rule-based детекторы;
  - LLM-детектор и генерация комментариев/предложений.
- Normative Checker: сверка по открытой нормативной информации через web-поиск.
- Local RAG Checker: сверка с локальными документами проекта.
- Exporter: JSON report + `docx` comments.
- Observability: метрики качества, latency, token/cost логи, ошибки по этапам.

### 5.2 Интеграции

- LLM provider: подключаемый интерфейс (первично OpenRouter/free models).
- Web search для нормативной сверки (без платного API на старте).
- Локальное файловое хранилище для PoC-артефактов.

## 6. Потенциальный data flow

1. `POST /api/upload`: загрузка -> парсинг -> `document_state` (параграфы + метки).
2. `POST /api/analyze/{id}`: построение полного текста -> анонимизация -> `anonymized_text + spans` -> анализ -> проверка на соответствие нормам и локальной базе -> формирование `analysis_report` (findings + anchors + evidence) -> сохранение в БД анонимизированного текста и отчета.;

3. `GET /api/document/{id}` экспорт изначального `docx` с комментариями.

### 6.1 Что делегируется LLM/Agent

- Классификация риск-фрагментов и формирование объяснения/редакционных предложений.
- Сопоставление формулировок с найденными нормами/локальными политиками (с обязательной ссылкой на evidence).

### 6.2 Что НЕ делегируется LLM

- Парсинг структуры документа.
- Анонимизация PII и контроль утечек.
- Политики хранения, логирование, token/cost логирование.
- Контроль безопасности tool-calls и интеграций.
