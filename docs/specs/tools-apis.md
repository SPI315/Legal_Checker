# Spec: Tools / APIs

## Назначение

Этот модуль описывает внешние и внутренние инструменты, через которые система выполняет анализ, поиск, OCR и другие вспомогательные операции. Все такие вызовы проходят через оркестратор, поэтому бизнес-логика не зависит напрямую от конкретного провайдера или реализации.

Важно различать два класса ошибок:

- ошибки входного файла и ingest-этапа;
- ошибки внешних инструментов (`LLM`, `web search`, `OCR`).

Ошибки входной валидации не должны проваливаться в downstream tool-calls. Если пользователь загрузил неподходящий файл, пайплайн останавливается до запуска OCR, retrieval и LLM.

## Input Validation / Ingest

На вход стандартного сценария принимаются только поддерживаемые форматы юридических документов, например `docx` и `pdf`.

Возможные ошибки:

- `unsupported_file_format` - файл не относится к разрешенным форматам;
- `document_corrupted` - файл формально поддерживается, но поврежден;
- `parse_failed` - структуру документа не удалось извлечь в ожидаемом виде.

Правило обработки:

- ошибки этого класса являются pre-tool ошибками и возвращаются пользователю без запуска OCR, retrieval и LLM stages.

## LLM API

LLM используется для интерпретации риск-фрагментов, генерации объяснений и предложений по правке.

Контракт вызова:

- метод: `analyze_risks(context, evidence, schema)`
- на вход передаются:
  - `context` - собранный контекст документа;
  - `evidence` - подтверждающие фрагменты из retrieval;
  - `response_schema` - ожидаемая схема ответа;
  - `model_config` - параметры модели.

Ожидаемый результат:

- JSON-объект строго по схеме;
- findings с `paragraph_id`, `evidence`, `confidence`, `suggested_edit`.

Ограничения и ошибки:

- timeout: `20s`;
- retry: до `2` попыток с exponential backoff;
- возможные ошибки:
  - `timeout`
  - `rate_limit`
  - `provider_5xx`
  - `invalid_json`
  - `policy_blocked`

## Normative API / Web Search

Этот инструмент используется для поиска нормативных источников и внешнего подтверждающего контекста.

Правила работы:

- разрешены только домены из allowlist;
- timeout: `8s`;
- cache TTL: `15 min`.

Возможные ошибки:

- `unavailable`
- `bad_source`
- `empty_result`
- `timeout`

## OCR API / Engine

OCR нужен для документов, где текст нельзя надежно извлечь обычным парсингом, прежде всего для PDF-сканов.

OCR не является механизмом приема произвольных пользовательских изображений как полноценного входного формата. Он используется только как внутренняя стадия для допустимых документов, прежде всего PDF-сканов или страниц, извлеченных из поддерживаемого контейнера документа.

Контракт:

- на вход подается бинарный файл, страница или изображение;
- на выходе система получает извлеченный текст и признаки качества распознавания.

Возможные ошибки:

- `ocr_failed`
- `low_quality_text`
- `unsupported_format`

## Side Effects

Интеграции не должны выполнять внешние действия, которые меняют состояние сторонних систем. Разрешены только:

- запись технических метрик;
- запись финального отчета;
- обновление локального индекса в отдельном процессе индексации.

Запрещены:

- любые внешние write-операции;
- отправка неанонимизированного текста во внешний LLM/API.

## Защита

Для всех инструментов действуют общие правила безопасности:

- tool allowlist на уровне orchestrator;
- валидация всех tool outputs по JSON schema;
- sanitization query для web-интеграций;
- обязательная privacy-проверка перед внешним LLM-вызовом.
