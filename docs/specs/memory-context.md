+# Spec: Memory / Context

## Session State

Session state нужен для управления жизненным циклом обработки одного документа. Он позволяет оркестратору понимать, на каком этапе находится запрос и можно ли его безопасно продолжить после ошибки.

Основные поля:

- `session_id`
- `stage`
- `status`
- `retry_count`
- `started_at`
- `updated_at`
- `degraded_flags`

TTL для PoC: `24 часа`.

## Memory Policy

Память делится на два уровня.

`Working memory` живет только во время активной обработки и содержит:

- структуру документа;
- OCR output;
- anonymized text;
- промежуточные findings;
- retrieval evidence;
- LLM drafts.

`Persistent memory` сохраняется после завершения обработки и содержит:

- final report;
- trace ids;
- stage metadata;
- model info;
- агрегированные метрики.

Полный исходный текст документа в persistent memory не сохраняется.

## Privacy / Data Policy

Перед внешним LLM-вызовом система формирует `anonymized_text`.

Допускается хранение:

- `anonymized_text`
- `spans`
- `analysis_report`
- технических метрик

Оригинальный текст разрешен только in-memory на время обработки.

## Context Budget

Контекст нужен для того, чтобы передавать в модель только релевантные части документа и не превышать лимит токенов.

Ограничение контекста в этом PoC связано не только с техническим лимитом модели, но и со стоимостью обработки. Для юридических документов один риск-кандидат требует не только фрагмент самого договора, но и подтверждающий контекст из нормативных источников. Поэтому budget policy управляет сразу тремя вещами: качеством grounding, укладыванием в контекстное окно и предсказуемостью стоимости на страницу и документ.

Основные правила:

- hard cap на один LLM-вызов: `32k` токенов на вход и выход вместе;
- приоритет включения:
  1. paragraph с candidate risk
  2. top evidence from retriever
  3. соседние paragraphs
  4. служебные инструкции

Если бюджет превышен:

- сначала отбрасываются низкоранговые evidence;
- затем сокращается соседний контекст;
- paragraph-источник риска сохраняется всегда.

С точки зрения стоимости в PoC используется простой ориентир: `до 1 ₽ на страницу` исходного документа.

Этот лимит задается как продуктовая верхняя граница, а не как точная финансовая модель. Он нужен, чтобы ограничивать число downstream LLM-вызовов, объем нормативного контекста и глубину сверки по одной странице или связанному с ней risk-кандидату.

Для документа на `100` страниц это дает ориентир `до 100 ₽` за полный прогон. Для документов меньшего объема бюджет масштабируется линейно по числу страниц.

Стоимость на страницу включает:

- анализ фрагмента самого документа;
- соседний контекст, если он нужен для корректной интерпретации формулировки;
- релевантные выдержки из законов и локальной базы;
- один или несколько downstream LLM-вызовов, если без этого нельзя надежно сравнить формулировки и собрать evidence.

Если в ходе обработки становится видно, что бюджет на страницу превышается, orchestrator должен сокращать cost через context budget:

- уменьшать число evidence-фрагментов;
- сокращать соседний контекст;
- ограничивать число повторных вызовов по одному risk-кандидату;
- переводить часть проверки в degraded mode, если это допустимо по качеству результата.

Цифра `1 ₽ на страницу` используется только как ориентир для PoC и подлежит уточнению после фактических замеров на реальных документах, моделях и провайдерах.
