# System Design: Legal Doc Checker PoC

## 1. Ключевые архитектурные решения

Система строится как сервис-ориентированная конвейерная архитектура с центральным оркестратором. Оркестратор управляет последовательностью шагов анализа, передает данные между сервисами и контролирует переходы между стадиями, повторы вызовов, fallback-сценарии и условия завершения обработки.

Архитектура включает несколько логических слоев:

- клиентский слой: frontend UI и backend API;
- прикладной слой: ingest, OCR, анонимизация, orchestration, risk analysis, evidence resolution;
- слой знаний: нормативный retrieval и локальный project RAG;
- инфраструктурный слой: storage, config, observability;
- внешние интеграции: LLM provider и нормативные источники.

Взаимодействие сервисов построено по модели ограниченных контрактов: каждый сервис получает строго определенный вход, выполняет одну ограниченную функцию и возвращает структурированный результат, который становится входом следующего этапа. Такой подход упрощает замену отдельных модулей, делает систему наблюдаемой и снижает связность между компонентами.

Входом системы является юридический документ в поддерживаемом формате, а также параметры обработки, например юрисдикция и тип анализа.

Для пользователя система должна выглядеть как единый сценарий: документ загружается один раз и сразу поступает в полный контур обработки без необходимости отдельно вызывать парсинг и анализ.

При этом внутренняя логика после этапа первичного выделения risk-кандидатов не является полностью линейной. В PoC используется минимальный agent loop внутри orchestrator: система последовательно выбирает следующий шаг анализа для каждого risk-кандидата, вызывает нужный tool/API, обновляет state и решает, достаточно ли собранного контекста для перехода к следующей стадии или требуется дополнительный retrieval, retry, fallback либо завершение.

Для `pdf`-сканов в контур обработки включается локальный OCR.

Перед внешними LLM-вызовами чувствительные данные проходят через контур анонимизации.

Выходом системы является структурированный отчет с найденными рисками, привязкой к абзацам, evidence, уровнем уверенности и предложением правки. Дополнительно формируются данные для UI и экспортные артефакты.

## 2. Модули и их роли

- `Frontend UI`: загрузка документа, запуск анализа, просмотр рисков, экспорт отчета.
- `API Gateway (FastAPI)`: входные контракты, валидация, маршрутизация, request id.
- `Orchestrator`: управление стадиями пайплайна, переходами, retry/fallback, stop condition.
- `Document Ingestion`: парсинг `.docx` и `.pdf`, сегментация, нормализация, offsets.
- `OCR модуль`: локальное извлечение текста из сканов PDF и оценка качества распознавания.
- `PII Anonymizer`: маскирование чувствительных данных перед внешними LLM/API-вызовами.
- `Risk Rules Engine`: детект типовых рисков по шаблонам и правилам.
- `Retriever Core`: запросы к нормативному контуру и локальному проектному индексу.
- `LLM Analyzer`: интерпретация риск-фрагментов, объяснение, предложение правки.
- `Evidence Resolver`: привязка findings к paragraph anchors и источникам.
- `Storage`: session state, reports, retrieval index metadata.
- `Observability`: логи этапов, latency, error rate, token/cost, eval hooks.

## 3. Основной workflow

С точки зрения пользовательского сценария основным входом является единый endpoint обработки документа, например `POST /api/documents/process`. Он принимает файл и сразу запускает полный пайплайн обработки.

Внутри системы этот запрос проходит следующие стадии:

1. прием файла и создание `session_id`;
2. ingest и структурирование документа;
3. для сканов PDF выполняется локальный OCR;
4. запуск оркестратора;
5. `Anonymization stage`: маскирование PII перед внешними вызовами;
6. `Rules stage`: быстрые rule-based кандидаты рисков;
7. `Retrieval stage`: нормативные и проектные evidence-контексты;
8. `LLM stage`: уточнение риска, объяснение, draft edit, confidence;
9. `Validation stage`: schema validation, citation checks, evidence checks, policy checks;
10. `Report stage`: формирование финального JSON-отчета и данных для UI;
11. `Export stage`: опциональная выгрузка артефактов, включая JSON и `docx` с комментариями на MVP-уровне.

Отдельные операции парсинга и анализа могут существовать как внутренние или технические endpoint-ы для отладки, повторного запуска отдельных стадий и интеграционных тестов, но не являются основным пользовательским сценарием.

После `Rules stage` фиксированный workflow переходит в ограниченный agent loop для каждого найденного risk-кандидата. Этот цикл не является open-ended автономным агентом и не планирует произвольные действия, но позволяет orchestrator принимать локальные решения внутри заданных границ пайплайна.

Минимальный agent loop выглядит так:

1. выбрать следующий risk-кандидат из state;
2. решить, нужен ли retrieval для этого кандидата и какие источники использовать: нормативный поиск, project RAG или оба;
3. выполнить retrieval и собрать evidence;
4. оценить, достаточно ли evidence и укладывается ли рабочий контекст в budget policy;
5. если evidence недостаточно, выполнить один дополнительный retrieval-pass с уточненным query или перейти в degraded mode;
6. если evidence достаточно, вызвать LLM для интерпретации риска и предложения правки;
7. провалидировать результат;
8. решить, можно ли принять finding, нужен ли retry, fallback в `rules-only` или переход к следующему risk-кандидату.

Таким образом, retrieval в системе является не самостоятельным сценарием, а инструментом внутри agent loop. Агентность в этом PoC ограничена выбором следующего действия в пределах заранее заданного workflow, budget policy и набора разрешенных tool/API интеграций.

## 4. State / Memory / Context Handling

`State` используется для управления жизненным циклом обработки документа. На уровне сессии система хранит `session_id`, текущую стадию, статус выполнения, checkpoints, количество повторных попыток и технические признаки деградации пайплайна. Это позволяет оркестратору понимать, на каком шаге находится обработка, можно ли безопасно продолжить выполнение после ошибки и какой fallback должен быть применен.

`Memory` делится на краткоживущую и сохраняемую. 
Краткоживущая память используется только внутри активной обработки и содержит разобранную структуру документа, результаты OCR, анонимизированную версию текста, промежуточные findings, retrieval-кандидатов, ответы LLM и статусы валидации. 
Сохраняемая память хранит только то, что нужно для отчета, трассировки и повторного анализа: итоговый report, технические метрики, trace ids, сведения о модели и статусах этапов. 

Полный исходный текст не сохраняется в логах и постоянном хранилище; оригинал документа допустим только in-memory на время обработки.

В memory попадают несколько типов данных:

- данные документа: paragraphs, paragraph ids, offsets, chunk map;
- privacy-данные: anonymized text, spans, статус маскирования и признаки допустимости внешнего вызова;
- аналитические данные: rule-based кандидаты, retrieval evidence, LLM findings, confidence, validation flags;
- технические данные: stage status, retry counters, latency, token/cost, trace metadata.

`Context handling` отвечает за формирование рабочего контекста  для LLM-вызовов. Система не передает в модель весь документ целиком, а собирает контекст из релевантных фрагментов: целевого paragraph, соседних абзацев, найденных evidence и служебных инструкций. Контекст формируется динамически под конкретный риск или группу кандидатов и проходит через policy checks перед отправкой во внешний контур.

Ограничение контекста реализуется через budget policy. Для одного LLM-вызова задается жесткий лимит по токенам, а при превышении бюджета применяется сокращение контекста: сначала отбрасываются низкоранговые evidence, затем уменьшается объем соседних фрагментов, при этом приоритет сохраняется за paragraph-источником риска и наиболее релевантными подтверждающими источниками. Для retrieval и LLM используется chunking документа на фрагменты размером `600-900` токенов с overlap `80-120`, чтобы сохранить локальный смысл и не потерять границы формулировок.

## 5. Retrieval-контур

Контур retrieval разделен на два направления. Первый отвечает за поиск по внешним нормативным источникам. Второй работает с локальной проектной базой и позволяет сопоставлять анализируемый документ с внутренними шаблонами, политиками, ранее согласованными формулировками и связанными документами.

Поиск в retrieval-контуре строится как многоступенчатый процесс: сначала формируется запрос на основе риск-кандидата, типа риска, юрисдикции и контекста абзаца, затем система выполняет первичный поиск по ключевым словам и семантическим представлениям, после чего найденные результаты проходят rerank. 

В итоговый контекст попадают только наиболее релевантные фрагменты, чтобы не перегружать последующие стадии и сохранить качество grounding.

Для локального retrieval используется индекс по versioned chunks, чтобы любой найденный фрагмент можно было однозначно связать с конкретным документом, его версией и положением внутри источника. 

Для внешнего retrieval важны фильтры по доверенности источника, свежести данных и типу правового документа.

Выход retrieval-контура представляет собой структурированные evidence-объекты: идентификатор источника, фрагмент текста, ссылку на источник, оценку релевантности и временную метку. 
В дальнейшем эти объекты используются оркестратором, LLM и валидаторами как единый слой подтверждающего контекста.

## 6. Tool / API интеграции

Все внешние вызовы инкапсулируются в отдельные адаптеры и вызываются через оркестратор, который контролирует порядок вызова, политику безопасности, таймауты, повторные попытки и обработку ошибок.

Основной внешний инструмент в системе - LLM provider API. Он используется для интерпретации риск-фрагментов, генерации объяснений и предложений по правке. 

Так как этот вызов является наиболее чувствительным и дорогим, он должен работать только через строго типизированный контракт ответа, с ограничением по времени выполнения, контролем повторных попыток и последующей валидацией результата.

Второй тип интеграций связан с нормативным поиском. Здесь используются web search или специализированные API, но только в рамках доверенных доменов. Это ограничение снижает риск попадания в контекст нерелевантных или ненадежных источников и соответствует требованиям governance по контролю внешних инструментов.

Для локального knowledge-контура используется векторное или гибридное хранилище индекса, которое обновляется только в процессе отдельной индексации. 

OCR-компонент, в свою очередь, проектируется как local-first модуль: внешние OCR/VLM-сервисы не используются.

## 7. Failure Modes, Fallback и Guardrails

При сбое одного из необязательных или внешних компонентов пайплайн по возможности не останавливается полностью, а переходит в деградированный режим с явной маркировкой ограничений результата. 
Например, при сбое retrieval анализ может быть продолжен с опорой на rules и LLM без внешнего нормативного подтверждения, а при недоступности LLM система должна вернуть хотя бы rule-based findings с пометкой о неполном анализе.

Критические сбои обрабатываются строже: 

- Если на вход подан файл неподдерживаемого формата, запрос отклоняется еще на этапе входной валидации с ошибкой `unsupported_file_format`. Например, одиночное изображение (`jpg`, `png`) не принимается как юридический документ для стандартного сценария анализа.

- Если документ не удалось корректно распарсить, пользователь получает ошибку этапа. 

- Если поддерживаемый файл поврежден или имеет некорректную внутреннюю структуру, это фиксируется как `document_corrupted` или `parse_failed`, и дальнейшие стадии не запускаются.

- Если OCR завершился ошибкой `ocr_failed` и текст недоступен другим способом, пользователь получает ошибку этапа.

- Если OCR дал низкое качество распознавания, это должно быть отражено в отчете как источник неопределенности. 

- Если retrieval недоступен, система может продолжить анализ в degraded mode с явной маркировкой ограничений результата.

- Если внешний LLM недоступен, превысил timeout или вернул невалидный ответ, orchestrator выполняет retry, а после исчерпания попыток завершает обработку в режиме `rules-only`, если это допустимо для текущего сценария.

- Если нарушена privacy-policy или не завершилась анонимизация, внешний LLM-вызов блокируется полностью, потому что в этом случае приоритетом становится не полнота анализа, а безопасность обработки.

Guardrails распределяются по всему пайплайну, а не только на этапе генерации ответа. 
На входе они включают валидацию формата файла, ограничение разрешенных источников и проверку политик обработки данных. 
Между сервисами guardrails реализуются через schema validation и явные межмодульные контракты. 
На этапе взаимодействия с моделью используются data-only policy для документа, защита от prompt injection и обязательная анонимизация перед внешними вызовами. 
На выходе применяется проверка структуры ответа, наличия evidence и запрет на категоричные юридические заключения без подтверждающего контекста.

## 8. Технические и операционные ограничения

Ключевое ограничение по производительности - целевой `p95 <= 60s` для документа среднего объема, порядка 15-20 страниц. 
Из этого напрямую следуют требования к многоступенчатому workflow: OCR, retrieval и LLM-вызовы не должны строиться как неограниченные по глубине циклы, а каждый внешний вызов должен иметь таймаут, ограничение числа повторов и понятный fallback. Архитектура должна обеспечивать завершение обработки даже в деградированном режиме.

С точки зрения затрат в PoC не вводится жесткий лимит стоимости на документ, но обязателен детальный token/cost учет по каждому LLM-вызову. 
Это означает, что система должна собирать стоимость не как вспомогательную метрику, а как часть базовой наблюдаемости. 
Такой подход позволит уже на этапе PoC понять, какие стадии наиболее дороги и где требуется оптимизация контекста, retrieval или выбора модели.

С точки зрения надежности целевым ориентиром является высокая доля завершения сценария без ручного ретрая и предсказуемое поведение при сбоях внешних компонентов. 
Для backend API это означает необходимость стабильной обработки запросов, а для оркестратора - корректного контроля повторных попыток, восстановления после частичных сбоев и прозрачной маркировки деградированных результатов.

Отдельный класс ограничений связан с политикой данных. Полный исходный текст документа не должен попадать в логи и постоянное хранилище. 
В долговременном хранении допускаются только анонимизированный текст, spans, analysis report и технические метрики. 
Оригинальный текст разрешен только in-memory на время выполнения обработки. 
Все секреты и параметры доступа к внешним сервисам должны передаваться через конфигурацию и переменные окружения, а не в коде или логах.
